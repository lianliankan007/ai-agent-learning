# Day4 学习总结: Agent Memory（长期记忆系统）

## 1. 为什么 Agent 需要 Memory
普通 LLM 更像“只看当前输入”的助手：
- 能处理当前 prompt
- 能利用当前上下文窗口里的对话
- 但不能稳定记住跨会话的信息

这会导致几个典型问题：
- 用户偏好记不住，比如“以后都用中文回答”
- 长任务进度会丢，比如“上次做到哪一步”
- 旧结论无法复用，比如“某个项目已经确定使用 Qdrant”

所以 Agent Memory 的核心作用，是给 Agent 增加跨轮次、跨会话的长期记忆能力。

可以把记忆分成三类：
- 短期记忆：当前会话历史，通常直接放在 messages 中
- 长期记忆：高价值信息写入数据库，后续按需检索
- 工作记忆：任务执行中的临时计划、工具结果和中间状态

## 2. 向量数据库的基本原理
Agent Memory 常见实现依赖“Embedding + 向量检索”。

完整链路如下：
1. 将文本转成向量，称为 Embedding
2. 将向量和元数据一起写入向量数据库
3. 查询时把用户问题也转成向量
4. 用相似度算法找出语义最接近的历史记忆
5. 将检索结果注入 prompt，辅助大模型作答

### Embedding 是什么
Embedding 就是把一段文本映射成一个高维数字向量。

例如：
- “我喜欢中文回答”
- “用户偏好中文输出”

虽然字面不同，但语义接近，所以 embedding 后两个向量的距离通常会比较近。

### 相似度检索是什么
常见算法包括：
- 余弦相似度
- 点积
- 欧氏距离

本次实现里使用的是余弦距离，对应 Qdrant 的 `Distance.COSINE`。

## 3. 记忆检索策略
### 3.1 简单检索
只做语义召回：
- query -> embedding
- 向量库 top-k

优点：
- 实现最简单
- 适合 Demo 和原型

缺点：
- 容易召回“语义接近但不该用”的信息

### 3.2 过滤检索
在语义召回基础上增加元数据过滤，例如：
- `user_id`
- `memory_type`
- `topic`

优点：
- 精度更高
- 支持多用户隔离
- 更适合实际项目

### 3.3 混合检索
把多种信号组合起来：
- 向量相似度
- 关键词重排
- importance 权重

本次代码中的混合检索做法是：
1. 先用向量召回更大范围候选
2. 再做关键词重叠评分
3. 最后与 importance 做加权排序

- 自动创建 collection

职责：
- 维护短期对话历史
- 检索长期记忆
- 把长期记忆拼接到 system prompt
- 调用 LLM 生成回答
- 按规则抽取高价值信息并自动写入长期记忆

自动抽取规则当前覆盖：
- `请记住...`
- `我叫...`
- `我喜欢...`
- `我不喜欢...`
- `我正在学习...`
- `我的项目是...`

提供命令行交互，方便演示三种检索模式：
- `mode simple`
- `mode filtered`
- `mode hybrid`

以及手动写入和检索：
- `remember 我喜欢中文回答`
- `remember-as preference style 用户偏好结构化输出`
- `search 我喜欢什么样的输出`
- `searchf preference 输出风格`

## 5. 当前实现的完整流程
一次完整对话的执行流程如下：

1. 用户输入问题
2. Agent 把问题转成 query embedding
3. 从 Qdrant 检索相关长期记忆
4. 把记忆整理成 memory context
5. 与短期对话历史一起发给 LLM
6. 生成回复
7. 如果用户输入命中了“值得记忆”的规则，再写入长期记忆

这就是一个最小可运行的 Agent Memory 闭环。

## 6. 实践中的关键经验
### 6.1 不是所有内容都应该存
长期记忆应该重点保存：
- 用户偏好
- 长期目标
- 项目事实
- 稳定约束
- 高价值结论

### 6.2 检索必须带约束
如果不按 `user_id`、`memory_type` 等维度过滤，很容易出现：
- 多用户数据串扰
- 旧记忆误召回
- 语义相近但业务不相关

### 6.3 混合检索更接近真实系统
单纯向量检索适合起步，但生产里通常需要：
- filter
- keyword
- rerank
- importance
