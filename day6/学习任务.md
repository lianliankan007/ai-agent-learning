# Day6 学习任务

## 学习主题

Agent Planning 与 Observation（让 Agent 学会“先判断，再行动，再观察”）

## 今日学习目标

今天的目标是把你在 Day5 学到的 Agent Loop 再往前推进一步：

1. 理解为什么 Agent 不应该一上来就盲目调用工具
2. 理解 Planning、Action、Observation 三者在 Agent 中分别负责什么
3. 实现一个“会先判断是否需要工具”的简化版 Agent
4. 学会通过日志观察 Agent 的每一步决策过程

学完今天后，你应该能解释：

1. 什么是 Agent 的 Planning
2. 什么是 Observation，为什么工具结果本质上就是 Observation
3. 为什么“会调用工具”不等于“会完成任务”
4. 如何让 Agent 在有限步数内更稳定地完成任务

## 理论知识

### 1. 什么是 Planning

在 Day5 中，你已经看到了 Agent Loop 的基本形态：

1. 用户提问
2. 模型决定下一步
3. 如果需要工具，就调用工具
4. 把结果放回上下文
5. 继续判断
6. 直到输出最终答案

但一个更成熟的 Agent，通常不会“看到问题就立刻调工具”，而是会先做一个很短的思考：

- 用户到底想解决什么问题
- 这个问题需不需要工具
- 如果需要，先调哪个工具
- 拿到结果后还缺什么信息

这个“先判断下一步做什么”的能力，就可以理解为最小版本的 Planning。

注意：

- 今天我们不是做复杂的任务分解器
- 而是先做一个教学版的“轻量 Planning”

也就是：

- 要么直接回答
- 要么先选工具
- 要么在得到 Observation 后继续下一步

### 2. 什么是 Action

Action 就是 Agent 真正执行的动作，例如：

- 调用 `get_current_time`
- 调用 `get_weather_by_context`
- 输出最终答案

也就是说，Planning 决定“做什么”，Action 负责“真正去做”。

### 3. 什么是 Observation

Observation 就是 Agent 执行动作之后得到的反馈信息。

最常见的 Observation 就是工具返回结果，例如：

- 当前时间是多少
- 某个城市天气如何
- 某个检索接口返回了什么内容

为什么 Observation 很重要？

因为 Agent 的下一步决策，不是凭空产生的，而是要根据上一步得到的新信息来继续判断。

可以这样记：

- Planning：想下一步
- Action：做下一步
- Observation：看结果

### 4. 为什么很多 Agent 会“乱调工具”

初学者常见问题是：

1. 工具描述不清楚
2. system prompt 没说清楚何时直接回答、何时查工具
3. 没有清晰的退出条件
4. 日志太少，看不出模型为什么这么决策

所以工程上要做好这几件事：

- 给工具写清晰描述
- 给模型明确规则
- 限制 `max_steps`
- 打印足够清楚的执行日志

### 5. 今天的重点不是“更强”，而是“更稳”

一个好的教学型 Agent，不一定一步到位做很多事，但必须满足：

- 结构清楚
- 日志可观察
- 行为可解释
- 失败时能知道卡在哪一步

这也是你后面做 Memory、RAG、完整 Agent 项目时非常重要的工程习惯。

### 任务2：实现一个“轻量 Planning Agent”

建议你实现一个类，例如：

`PlanningAgentDemo`

至少包含以下结构：

- `messages`
- `tool_definitions`
- `tool_registry`
- `max_steps`
- `run()`
- `_call_api()`
- `_execute_tool()`

和 Day5 的区别是：

- Day5 更偏“进入循环后按步骤执行”
- Day6 更强调“每一步先判断，到底是直接回答，还是调用工具”

### 任务3：把工具链拉长到 4 到 6 步

建议至少实现下面 4 个工具，如果你状态不错，可以直接做到 6 个：

- `get_current_location(user_name: str)`
- `get_current_time(city: str)`
- `get_weather_by_context(city: str, current_time: str)`
- `get_clothing_advice(weather: str, current_time: str)`
- `estimate_clothing_layers(clothing_advice: str)`
- `get_clothing_weight(clothing_advice: str, layer_count: str)`

要求：

- 先用假数据实现即可
- 工具函数要有简短中文注释
- 参数名尽量简单、清楚
- 工具之间要能形成明显的前后依赖关系

### 任务4：让 Agent 支持三类问题

你的 Agent 至少要正确处理下面两种输入：

1. 不需要工具的问题
例如：

- `你是谁？`
- `请解释什么是 Agent Loop`

2. 需要短链工具的问题
例如：

- `北京现在几点？`
- `上海天气怎么样？`
- `北京现在几点，并告诉我天气`

3. 需要长链规划的问题
例如：

- `请先查小明在哪个城市，再告诉我当前时间和天气`
- `请帮我查询小明的位置、时间、天气，再给出穿衣建议`
- `请帮我完成位置 -> 时间 -> 天气 -> 穿衣建议 -> 层数 -> 重量 这一整条链路`

核心目标是让你观察：

- 有的问题应该直接回答
- 有的问题才应该调用工具
- 有的问题虽然需要工具，但也不一定要把整条链都走完

### 任务5：补充清晰日志

程序运行时，至少打印这些信息：

- 当前是第几步
- 用户问题是什么
- 模型本轮是直接回答，还是调用工具
- 如果调工具，调用了哪个工具
- 工具参数是什么
- Observation 是什么
- 是否已经结束循环

### 任务6：加一个简单的退出策略

要求：

- 当模型直接返回普通文本且没有工具调用时，结束循环
- 当达到 `max_steps` 时，主动停止
- 返回一个清晰的失败提示，告诉用户是因为达到最大步数而停止

### 任务7：做一次对比实验

至少测试这三类问题，并把现象写进学习总结：

1. 纯聊天问题
2. 单工具问题
3. 多工具串联问题

重点观察：

- Agent 是不是每次都查工具
- 哪些问题其实不应该查工具
- 多工具问题是否比单工具问题更容易超步数
