# Day6 学习总结

## 今日学习目标

1. 理解 Planning / Action / Observation 这三个概念在 Agent 中分别负责什么
2. 在 Day5 的 Agent Loop 基础上，继续强化“先判断，再行动，再观察”的思维
3. 实现一个带长链工具调用能力的 Planning Agent
4. 学会通过日志分析 Agent 的每一步决策过程

## 学习内容总结

今天的核心不是换了一套全新的 Agent 架构，而是在 Day5 的 Agent Loop 骨架上，进一步理解 Agent 每一步到底在做什么。

从代码结构上看，Day6 仍然是典型的 Agent Loop：

1. 组装 `messages`
2. 调用 LLM
3. 判断是否有 `tool_calls`
4. 如果有，就执行工具
5. 把工具结果作为新的上下文放回去
6. 继续下一轮，直到输出最终答案或达到 `max_steps`

也就是说，Day6 和 Day5 在主流程上没有本质换框架，仍然都是 Loop。

但 Day6 比 Day5 更强调“如何理解这个 Loop”：

- `Planning`：模型先判断下一步该做什么
- `Action`：真正调用工具，或者直接输出答案
- `Observation`：读取工具返回结果，再决定下一步

所以 Day6 的重点不只是“让 Loop 跑起来”，而是“让 Loop 变得更可解释”。

另外，今天还把 Day6 的调用链从短链扩展成了长链，参考 Day5 补成了 6 步工具链：

1. `get_current_location`
2. `get_current_time`
3. `get_weather_by_context`
4. `get_clothing_advice`
5. `estimate_clothing_layers`
6. `get_clothing_weight`

这样做的意义是：

- 可以更清楚地观察多步决策
- 可以更明显地看到 Observation 如何影响下一步
- 可以理解为什么 `max_steps`、prompt、工具描述都非常重要

## 核心知识点

### 1. Day6 本质上仍然是 Agent Loop

Day6 不是新的 Agent 架构，而是对 Day5 Agent Loop 的认知强化。

如果只看代码主流程，Day5 和 Day6 的共同点是：

- 都有 `messages`
- 都有 `tool_definitions`
- 都有 `tool_registry`
- 都有 `run()`
- 都是循环调用模型
- 都是把工具结果塞回上下文

所以可以说：

“Day6 在代码骨架上还是 Day5 那套 Agent Loop，只是在同一个 loop 里，强化理解 Planning / Action / Observation 三个角色。”

### 2. Planning 不等于复杂任务分解

今天理解到的 Planning，不是那种很复杂的“自动生成完整计划树”。

在当前教学 demo 里，Planning 更接近：

- 这个问题需不需要工具
- 需要的话，先调哪个工具
- 工具结果拿到后，是否还需要继续
- 什么时候应该停下来直接回答

这是一种最小版本的 Planning。

### 3. Observation 本质上就是“新信息”

工具返回结果之所以重要，是因为 Agent 的下一步不是凭空决定的，而是要根据新拿到的信息继续判断。

例如：

- 先拿到位置
- 再根据位置查时间
- 再根据位置和时间查天气
- 再根据天气和时间给穿衣建议

这个过程里，每一步工具结果都属于 Observation。

### 4. 长链任务更能暴露 Agent 的问题

长链任务比短链任务更容易暴露这些问题：

- 工具顺序是否合理
- 参数是否传对
- prompt 是否清晰
- 是否会过度调用工具
- 是否会停不下来

所以 Day6 把链路拉长以后，更适合拿来观察 Agent 的稳定性。

### 5. `max_steps` 仍然很重要

即使 Day6 强调的是 Planning / Action / Observation，工程上仍然要依靠 `max_steps` 做保护。

因为模型可能会：

- 重复调用工具
- 错误理解任务
- 在信息已经足够时继续多走几步

所以 `max_steps` 不是教学装饰，而是实际开发中很重要的防护手段。

## 遇到的问题

### 1. 容易误以为 Day6 是全新的架构

一开始容易把 Day6 理解成“新的 Agent 代码模式”，但后面分析后发现，它本质上还是 Day5 的 Agent Loop。

真正变化的是理解角度，而不是主流程框架。

### 2. 容易把“会多步调用工具”和“会规划”混为一谈

如果只看表面现象，会觉得：

- 能连续调多个工具 = 会 Planning

但其实不是。

如果模型只是机械地把链路走完，不考虑是否真的有必要继续，那它只是“会执行长链”，不代表它真的有决策能力。

### 3. 长链更容易超步数

工具链一旦拉长，就更容易遇到：

- 某步参数错误
- 模型不及时停下
- 多余调用后续工具

这也说明 Day6 比 Day5 更适合拿来观察 Agent 的稳定性问题。

## 解决方案

### 1. 从“代码骨架”和“概念重点”两个层面区分 Day5 / Day6

我最后采用的理解方式是：

- 从代码骨架看：Day5 和 Day6 都是 Agent Loop
- 从教学重点看：Day5 重执行框架，Day6 重决策解释

这样就不容易混淆。

### 2. 通过长链工具让 Observation 更明显

如果只有两个工具，Planning / Action / Observation 的区别还不够明显。

把链路拉长之后，可以更清楚看到：

- 前一步的 Observation 如何成为下一步的输入
- 为什么 Agent 是一个连续决策过程

### 3. 用日志帮助理解 Agent 行为

日志不是为了“看着热闹”，而是为了帮助分析：

- 这一步为什么调了这个工具
- 上一步结果是否真的影响了下一步
- 是否已经具备最终回答所需的信息

这对后面做 RAG 和完整 Agent 项目也会非常重要。

## 今日收获

1. 我真正理解了 Day6 和 Day5 的关系：代码结构上差别不大，但 Day6 更强调对 Agent 决策过程的解释。
2. 我理解了 Planning / Action / Observation 不一定是复杂理论，它们可以直接映射到实际代码执行过程。
3. 我意识到 Agent 的难点不只是“把工具调起来”，而是“什么时候该调、什么时候该停”。
4. 我开始建立一种更工程化的视角：除了看能不能跑通，还要看行为是否稳定、日志是否清晰、退出条件是否合理。
