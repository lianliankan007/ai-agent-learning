# Day12 学习总结

## 今日学习目标
### 知识切分的常见方式

  - 固定长度切分
      - 按字数、token 数或句子数平均切。
      - 优点是实现简单、稳定。
      - 缺点是容易把一个完整语义硬切开。
  - 按自然结构切分
      - 按标题、段落、小节、列表、代码块来切。
      - 优点是更符合文档原本结构，语义通常更完整。
      - 缺点是不同段落长度差异可能很大，不够均匀。
  - 按语义切分
      - 根据内容主题变化来切，比如一个定义、一条规则、一个案例各自单独成块。
      - 优点是语义最纯。
      - 缺点是实现复杂，成本更高。
  - 混合切分
      - 先按自然结构切，再对过长段落按长度二次切分。
      - 这是实际项目里比较常见、也比较稳的做法。

  什么样的切分更适合 RAG
  不是切得越细越好，而是要满足两个条件：

  - 一个 chunk 最好只表达一个核心知识点
  - 一个 chunk 最好又能独立看懂，不依赖过多前后文

  也就是说，理想状态是：

  - 语义单一
  - 上下文完整

  为什么切分会影响 embedding 质量
  Embedding 模型会把整段文本压缩成一个向量。
  如果一段文本里混了很多主题，比如：

  - 第一段讲安装
  - 第二段讲原理
  - 第三段讲报错处理

  那模型生成的向量就会同时包含多种语义，结果是：

  - 表达不聚焦
  - 查询匹配不够准

  如果一个 chunk 只讲“HNSW 是什么”，那它的向量就更集中地表达“HNSW 原理”这个主题，检索时更容易被命中。

  为什么切分会影响最终回答质量
  RAG 不是“检索完就结束”，后面还要把召回内容交给 LLM。
  所以切分不仅影响“能不能找到”，还影响“模型能不能用好”。

  如果 chunk 切得好：

  - 检索结果更精准
  - 传给 LLM 的上下文噪声更少
  - 模型更容易抽取关键信息
  - 回答更准确、更稳定

  如果 chunk 切得不好：

  - 检索结果可能偏题
  - 召回内容可能缺前因后果
  - LLM 容易误解、脑补或答非所问

  一个简单例子
  原文：

  “Redis 支持多种数据结构。字符串适合缓存对象。列表适合消息队列。集合适合去重。Redis 也支持持久化，包括 RDB 和 AOF。”

  如果整段作为一个 chunk：

  - 这个 chunk 同时讲了数据结构用途和持久化
  - 用户问“Redis 的 AOF 是什么”
  - 检索到的内容里会混入很多无关信息

  如果切成两个 chunk：

  - chunk1：Redis 数据结构及其适用场景
  - chunk2：Redis 持久化机制，RDB 和 AOF

  那问 “AOF 是什么” 时，更容易命中 chunk2。

  为什么要做 overlap
  overlap 就是相邻 chunk 之间保留一点重复内容。

  原因是：

  - 关键信息可能正好落在切分边界
  - 如果完全硬切，某个 chunk 可能缺开头，另一个 chunk 缺结尾
  - 加一点重叠，能让上下文更连续

  例如：

  - chunk1: 第 1 到 200 token
  - chunk2: 第 180 到 380 token

  这样 180 到 200 这部分会重复一次，减少信息被切断的风险。

  overlap 太大也有问题

  - 冗余数据变多
  - 索引体积更大
  - 检索结果可能返回多个高度重复的 chunk
  - LLM 上下文被重复内容占满

  所以 overlap 也不是越大越好，一般只是“补边界”，不是“大量重复”。

  如何判断 chunk 切得好不好
  可以看几个指标：

  - 检索命中率是否高
  - 召回内容是否足够回答问题
  - 召回结果是否经常包含大段无关内容
  - 同一个问题是否总是召回很多重复 chunk
  - LLM 是否经常因为上下文不完整而答错

  简单说：

  - 命中了但答不好，可能 chunk 不完整
  - 经常命中偏题，可能 chunk 太大或太混
  - 经常返回很多重复片段，可能 overlap 或切分粒度不合适

  不同类型文档的切分建议

  - FAQ 文档
      - 按“一问一答”切分最好
  - 技术文档
      - 按标题、小节、代码示例切分
  - 法规/制度文档
      - 按条款、章节切分
  - 教程类文档
      - 按步骤、主题块切分
  - 聊天记录/日志
      - 按轮次、会话片段切分

  一句更完整的总结
  知识切分的本质，是把原始文档改造成适合向量表示、相似度匹配和 LLM 理解的检索单元。切分得合理，每个 chunk 既语义聚焦又信息完整，向量检索和最终问答效果都会明显更好。
