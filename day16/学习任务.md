## 理论知识

### 1. Tool Use 不只是“调用一下工具”

很多初学者会把 Tool Use 理解成：

- 选中工具
- 执行一下
- 把结果拿回来

但真实系统里，工具调用是否有效，往往取决于两个关键点：

1. 参数是不是传对了
2. observation 是不是被正确理解和使用了

如果参数不对，就算工具本身没问题，结果也可能完全错误。

### 2. 什么是工具参数

工具参数就是：

- 程序真正传给工具的输入

例如：

- 计算器工具的参数是表达式
- 搜索工具的参数是查询语句
- 查询天气工具的参数可能是城市名和日期
- 数据库工具的参数可能是 SQL 条件或过滤字段

参数设计不清晰时，常见问题包括：

- 参数缺失
- 参数格式不对
- 参数粒度太粗
- 参数里混入无关信息

### 3. 什么是 observation

observation 可以理解为：

- 工具执行后返回给 Agent 的结果

它不是最终回答，而是中间结果。

例如：

- calculator 返回 `166`
- search 返回 3 条文档
- memory lookup 返回一段历史信息
- API 工具返回 JSON 数据

真正的 Agent 不应该把 observation 生硬原样抛给用户，而是应该：

1. 先理解 observation
2. 判断它是否足够支撑回答
3. 再组织成对用户有用的输出

### 4. 为什么 Day16 很重要

如果 Day15 解决的是：

- 要不要调工具

那么 Day16 解决的是：

- 工具调了之后，怎么把它调对

这一步对后面做真正的 Function Calling Agent 非常关键，因为真正的系统常见错误并不是：

- 没调用工具

而是：

- 工具参数抽错了
- observation 理解错了
- 结果虽然拿回来了，但没有被正确利用
