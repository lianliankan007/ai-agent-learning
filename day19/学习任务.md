## 理论知识

### 1. 为什么真实 Agent 必须考虑工具失败

在教学 Demo 里，工具常常是稳定的。

但真实工程里，工具可能出现很多异常：

- 网络超时
- 接口限流
- 参数错误
- 权限不足
- 目标数据不存在
- 第三方服务临时不可用

如果 Agent 没有失败处理能力，就会出现两个严重问题：

1. 一旦失败，整个流程直接中断
2. 模型可能基于错误 observation 继续推理，导致误答

所以真正可用的 Agent，不只是“能调工具”，而是“失败了也知道怎么处理”。

### 2. 什么是可重试错误

可重试错误，通常是暂时性的、上下文没有变的错误。

常见例子：

- HTTP 429 限流
- 短暂超时
- 网络抖动
- 下游服务瞬时异常

这类错误的特点是：

- 不是因为参数本身错了
- 也不是因为目标根本不存在
- 过一会儿重试，可能成功

所以这类错误适合做有限次数的 retry。

### 3. 什么是不可重试错误

不可重试错误，通常不是“再试一次就会好”的问题。

常见例子：

- 城市名称根本不支持
- 参数为空
- 权限不够
- 业务规则禁止

这类错误如果还一直重试，只会：

- 浪费时间
- 浪费 token
- 增加系统负载
- 让 Agent 陷入假忙碌

所以不可重试错误更适合：

- 立即停止
- 尝试降级方案
- 或者明确告诉用户当前边界

### 4. 什么是重试与降级

重试（retry）是：

- 继续调用同一个工具
- 期待临时问题恢复

降级（fallback）是：

- 主工具失败后，切换到另一个更稳但能力更弱的工具
- 例如查实时接口失败后，改查缓存

所以它们解决的问题不同：

- retry 解决“暂时不可用”
- fallback 解决“主路径走不通，但还想尽量完成任务”

### 5. 为什么失败恢复常常要放在程序层

很多初学者会以为：

- 工具失败了，让模型自己再想想不就行了

但真实系统中，更可靠的做法通常是：

- 程序层先统一判断错误类型
- 决定是否重试
- 决定是否走 fallback
- 再把恢复结果作为 observation 给模型

原因是：

1. 重试次数需要可控
2. 降级策略需要稳定
3. 错误分类要可审计
4. 不能把所有控制权都交给模型随机决策

这也是 Day19 最重要的工程意识。
