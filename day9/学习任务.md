## 理论知识

### 1. Day8 的最小 RAG 解决了什么

Day8 你已经完成了最小 RAG 闭环：

用户问题 -> 检索知识 -> 拼接 prompt -> 调用 LLM -> 输出答案

这是非常重要的一步，因为你已经理解了：

- RAG 的本质是先找资料，再回答
- 模型不是凭空回答，而是基于外部上下文回答

但 Day8 的检索方式是关键词匹配，这种方式虽然适合入门，却有明显局限。

### 2. 关键词检索为什么不够

关键词检索的问题在于，它太依赖“字面是否一致”。

例如：

知识库里写的是：

- `Agent Loop 是一种循环执行机制`

用户问的是：

- `智能体为什么要循环决策？`

这两个句子表达的是相近意思，但如果没有出现完全相同的关键词，简单检索器就可能命不中。

所以关键词检索常见问题有：

- 同义词命不中
- 换一种说法就命不中
- 语义相近但字面不同的内容难以匹配
- 知识库一大，效果会明显下降

### 3. 什么是 Embedding

Embedding 可以理解成：

把一句话、一个词、一段文本，转换成一个“表示语义特征的向量”。

你可以暂时把它想成：

- 文本 -> 向量坐标

如果两段文本语义接近，它们转换后的向量距离通常也会比较近。

例如：

- `什么是 Agent Loop`
- `智能体为什么要循环执行`

虽然字面不完全一样，但语义相近，所以 embedding 后通常会比较靠近。

### 4. 什么是向量检索

向量检索的思路是：

1. 先把知识库文本都转成向量
2. 再把用户问题也转成向量
3. 比较问题向量和知识向量的距离
4. 找出最相似的几条知识

这比关键词检索更强，因为它不是只看“字面命中”，而是在找“语义上最像”的内容。

### 5. 向量检索为什么更适合 RAG

RAG 的关键不是机械地命中关键词，而是找到“真正相关”的资料。

而向量检索更适合做这件事，因为它更擅长处理：

- 同义表达
- 近义问法
- 自然语言变形
- 长文本的语义匹配

所以在真正的 RAG 项目里，常见流程通常是：

切分文档 -> embedding -> 存入向量库 -> 查询时做相似度检索 -> 把结果喂给 LLM

### 6. 今天为什么仍然不急着上完整向量库

因为你现在更需要先把概念吃透：

- 为什么关键词检索会失效 :关键词搜索匹配度查
- embedding 到底在解决什么问题：解决近似语法、近以问法、长文本雨衣匹配
- 为什么“相似度”比“是否命中关键词”更高级：通过多维向量的映射，找到相近的回答，比约束固定回答，等等要简单的多

如果这层没有吃透，后面即使你会调用 Qdrant、Milvus 或 LangChain，也容易只会照着教程敲。
