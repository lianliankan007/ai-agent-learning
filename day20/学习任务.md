## 理论知识

### 1. 为什么工具一多，系统会更容易出错

当系统只有 1 到 2 个工具时，路由问题还不明显。

但当工具数量增加后，模型常见错误会明显上升：

- 本该查天气，却调了知识检索
- 本该查日程，却去查长期记忆
- 本该直接回答，却强行调用工具
- 本该先查一个核心工具，却选了边缘工具

所以多工具系统的关键挑战，不是“工具多不多”，而是：

- 工具边界是否清晰
- 模型是否知道每个工具适用什么问题
- 程序层是否能对路由结果做最基本的校验

### 2. 什么是 Tool Routing

Tool Routing 可以理解成：

- 在多个候选工具中，为当前问题选择最合适的那一个

它本质上解决的是一个选择问题，而不是一个执行问题。

最小链路通常是：

1. 用户提问
2. 模型判断是否需要工具
3. 如果需要，再判断最适合哪个工具
4. 程序执行工具
5. 再根据 observation 继续决策或结束

所以 Tool Routing 是 Tool Use 的上游能力。

### 3. 多工具误路由的常见类型

常见误路由至少有 4 种：

1. 概念问题误路由到外部查询工具
2. 实时信息误路由到记忆工具
3. 日程问题误路由到知识检索工具
4. 本可直接回答的问题被过度工具化

这些问题的共同点是：

- 不是工具本身坏了
- 而是工具选择阶段就已经偏了

### 4. 为什么工具描述和边界特别重要

如果两个工具都写成：

- “用于查询信息”

那模型几乎无法稳定区分：

- 天气查询
- 日程查询
- 文档查询
- 记忆查询

所以真正可用的多工具系统，必须做到：

1. 每个工具都有清晰边界
2. 每个工具都写清适用场景
3. 参数名要体现业务语义
4. 工具之间尽量少重叠

### 5. 为什么程序层仍然要做校验

即使模型已经做了结构化 tool call，程序层也不能完全裸奔。

例如：

- 参数 key 写错了
- 参数名和工具定义不一致
- 模型虽然选对了工具，但 arguments 结构不稳定

如果程序层完全不兜底，系统很容易因为一个小偏差直接崩掉。

所以 Day20 的工程意识是：

- 路由主要由模型做
- 但边界约束、参数归一化、执行安全仍然要由程序控制
