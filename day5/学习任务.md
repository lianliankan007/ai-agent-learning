## 理论知识

### 1. 什么是 Agent Loop

普通的 LLM 调用流程通常是：

- 用户提问
- 程序调用一次模型
- 模型直接返回答案

但很多任务无法一步完成，比如：

- 先判断要不要查工具
- 再执行工具
- 再根据工具结果继续判断下一步
- 最后再输出最终答案

这时就需要 Agent Loop。

Agent Loop 本质上是一个循环控制机制：

1. 接收任务
2. 让模型决定下一步动作
3. 如果需要工具，就执行工具
4. 把工具结果放回上下文
5. 再次让模型决定下一步
6. 直到得到最终答案

### 2. Agent Loop 和 Function Calling 的关系

Day3 的 Function Calling 重点是：

- 模型能调用工具

Day5 的 Agent Loop 重点是：

- 模型能连续多步调用工具
- 程序能控制整个执行过程

所以可以这样记：

- Function Calling 是工具调用能力
- Agent Loop 是任务执行框架

### 3. Agent Loop 的最小结构

一个最小版本通常包括：

- `messages`：保存上下文
- `tools`：可调用工具列表
- `tool_registry`：工具名到函数的映射
- `max_steps`：最大循环步数
- `run()`：驱动整个执行循环

简化流程如下：

```text
用户问题
  ↓
LLM 判断是否需要工具
  ↓
如果需要工具 -> 执行工具
  ↓
把工具结果加入 messages
  ↓
再次调用 LLM
  ↓
直到模型输出最终答案
```

### 4. 为什么要限制最大步数

如果不限制步数，可能出现：

- 模型一直重复调用工具
- 参数不对导致反复重试
- Prompt 不清楚导致循环不退出

所以工程上必须设置：

- `max_steps=3` 或 `max_steps=5`

超过后程序应主动停止，并返回说明信息。

## 实践任务

### 任务1：实现本地工具

在 `day5/agent_loop_demo.py` 中实现至少两个工具：

- `get_current_time(city: str)`
- `get_weather(city: str)`

可以先使用假数据，不要求接真实天气 API。

### 任务2：实现最小 Agent Loop

要求包含：

- 系统提示词
- `messages` 上下文
- 工具定义
- 调用 LLM 判断下一步动作
- 执行工具
- 追加工具结果
- 再次调用 LLM
- 输出最终答案

### 任务3：加入最大步数保护

要求：

- 最多执行 3 步
- 超过就终止
- 给出明确提示

### 任务4：打印执行日志

程序运行时，至少输出如下信息：

- 当前是第几步
- 模型决定调用哪个工具
- 工具参数是什么
- 工具结果是什么
- 是否已经输出最终答案

建议你观察执行日志，理解 Agent 是如何一步步完成任务的。
