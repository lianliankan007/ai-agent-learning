 ## 1) LLM 参数深入：temperature / max_tokens / top_p

  - temperature：控制“随机性”
      - 低（0~0.3）：更稳定、保守，适合问答/总结/代码
      - 中（0.4~0.7）：平衡创造力与稳定性
      - 高（0.8+）：更发散，适合创意写作
  - top_p：核采样范围（概率质量）
      - 小（如 0.3）：只从高概率词里选，更稳
      - 大（如 0.9~1.0）：候选更广，更灵活
  - max_tokens：限制输出长度
      - 太小：回答被截断
      - 太大：成本上升 + 可能跑偏

  实战建议：

  - 先固定 top_p=1，只调 temperature
  - 任务型场景默认：temperature=0.2~0.5
  - 明确设置 max_tokens 防止超长输出

  ## 2) 多轮对话：上下文管理

  核心问题：模型“记忆”来自你每次发送的消息历史，不是真正长期记忆。

  建议结构：

  1. system：角色与规则（一次性原则）
  2. user/assistant：最近若干轮关键对话
  3. 必要时加“总结记忆”（压缩旧对话）

  常见策略：

  - 滑动窗口：只保留最近 N 轮
  - 摘要压缩：对旧历史做 summary，替代原文
  - 关键信息抽取：只保留用户偏好、约束、结论

  ## 3) 错误处理与重试

  你要区分“该不该重试”：

  - 应重试：
      - 网络超时、连接中断（临时故障）
      - 429 限流（等待后重试）
      - 5xx 服务端异常
  - 不应盲重试：
      - 400/401/403（参数、权限、密钥问题）

  推荐模式：

  - 指数退避（1s, 2s, 4s...）+ 随机抖动（jitter）
  - 设置最大重试次数（如 3~5 次）
  - 为每次请求设置超时
  - 记录日志：错误类型、重试次数、耗时、request id
