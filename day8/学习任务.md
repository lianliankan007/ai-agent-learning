## 理论知识

### 1. 为什么需要 RAG

大模型虽然很强，但它有几个天然问题：

- 知识可能过时
- 不知道你的私有资料
- 容易一本正经地胡说
- 对精确业务事实不稳定

例如：

- 公司内部制度
- 项目文档
- 接口说明
- 业务知识库

这些内容通常不在模型参数里，或者即使模型“似乎知道”，也不可靠。

这时候就需要 RAG。

RAG 的本质不是“训练一个更大的模型”，而是：

- 先从外部知识库中检索相关内容
- 再把检索结果放进 prompt
- 最后让模型基于这些资料回答

### 2. 什么是 RAG

RAG 全称是：

Retrieval-Augmented Generation

可以拆成两部分：

#### 1. Retrieval

先检索相关资料。

#### 2. Generation

再让模型根据检索到的资料生成答案。

所以 RAG 的核心链路可以写成：

用户问题 -> 检索相关资料 -> 拼接上下文 -> 调用 LLM -> 输出答案

### 3. RAG 和 Agent 的区别

这两个概念很容易混。

你可以这样记：

- RAG 解决的是“模型缺知识、缺上下文”的问题
- Agent 解决的是“模型如何连续决策、调用工具、完成任务”的问题

RAG 更像是在给模型“喂资料”。

Agent 更像是在让模型“做事”。

当然，真实项目里它们经常结合：

- Agent 可以把“检索知识库”当成一个工具
- RAG 可以成为 Agent 的一个子模块

### 4. 最小版本的 RAG 长什么样

今天先不追求复杂架构。

一个最小 RAG Demo 只需要这些部分：

1. 准备几段知识文本
2. 根据用户问题找到最相关的知识
3. 把知识拼进 prompt
4. 让模型基于知识回答

今天你先可以不做向量检索，先做一个教学版检索器，例如：

- 关键词匹配
- 简单包含判断
- 简单分数排序

重点不是检索算法高级，而是要先吃透 RAG 的主流程。

### 5. 为什么今天不急着上向量数据库

因为如果一上来就引入：

- embedding
- vector db
- top_k
- rerank

很容易把你注意力从“RAG 的核心思想”拉走。

今天更重要的是先明白：

- 为什么要检索
- 检索结果如何进入 prompt
- 没检索和有检索，回答有什么差异

等你把这个最小闭环吃透，后面再升级到 embedding + 向量检索会顺很多。

### 任务2：准备一个小型知识库

请你先手动准备 5 到 8 条知识文本。

建议做成列表，内容可以围绕 AI Agent 学习主题，例如：

- 什么是 Token
- 什么是 Context Window
- 什么是 Function Calling
- 什么是 Agent Loop
- 什么是 RAG
- 什么是 Memory

要求：

- 每条知识尽量短小清楚
- 先硬编码在脚本里即可
- 用中文内容，方便你观察效果

### 任务3：实现最小检索逻辑

要求：

- 不用向量数据库
- 不用 embedding
- 先实现一个简单的关键词匹配检索器

例如思路：

1. 用户问题分词或按关键词判断
2. 统计每条知识命中次数
3. 选出最相关的前 2 到 3 条

重点是让你理解：

- RAG 的第一步是先找资料
- 检索结果质量会直接影响最终回答

### 任务4：把检索结果拼进 Prompt

你的 Prompt 至少要包含这两部分：

1. 检索到的知识片段
2. 用户问题

并明确告诉模型：

- 优先基于提供的知识回答
- 如果知识不足，要明确说明
- 不要脱离资料乱编

### 任务5：与 LLM 交互时继续打印日志

要求：

- 继续使用 `llm_markdown_logger.py`
- 额外在控制台打印：
  - 用户问题
  - 检索到的知识
  - 最终拼接出来的 prompt 核心内容
  - 最终回答

### 任务6：做一组对比实验

至少测试下面两类问题：

1. 知识库中有答案的问题
2. 知识库中没有答案的问题

重点观察：

- 有资料时，模型回答是否更聚焦
- 没资料时，模型是否还能保持克制

### 任务7：思考如何从最小 RAG 升级到真正 RAG

今天不用实现，但要在总结里写出你的理解：

- 关键词检索的缺点是什么
- 为什么后面还要学 embedding 和向量检索
- 如果知识库变成几千条，今天的实现会遇到什么问题
