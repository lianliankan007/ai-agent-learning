## 理论知识

### 1. 为什么第 1 周要做复盘

很多同学学 Agent 时有一个常见问题：

- 每天都写了一点代码
- 每天都感觉懂了一些概念
- 但一旦让自己解释“这些模块到底是什么关系”，就开始混乱

所以 Day7 的复盘非常重要。

这一周你其实已经接触了 4 个层次：

1. 直接调用 LLM
2. 给 LLM 增加上下文管理
3. 让 LLM 调工具
4. 让 LLM 在循环里持续决策

如果这 4 层关系理顺了，后面学 RAG、Memory、完整 Agent 项目就会轻松很多。

### 2. 本周知识主线怎么串起来

你可以把这几天的内容理解成一条演进路线：

#### Day1 到 Day2

重点是：

- 如何调用 LLM
- 如何控制参数
- 如何维护对话历史

这时程序还只是“会聊天的 LLM 应用”。

#### Day3

重点是：

- 模型不只是回答，还能调用工具

这时系统开始具备“接触外部世界”的能力。

#### Day5

重点是：

- 模型可以连续多步调用工具
- 程序可以控制循环
- 系统不再是一次调用，而是一个执行过程

这时你开始接触真正的 Agent Loop。

#### Day6

重点是：

- 不是只会循环，还要会判断
- 每一步都要分清 Planning、Action、Observation

这时你开始从“能跑”进入“可解释”的 Agent。

### 3. 什么是 Agent 稳定性

一个 Agent 不是能跑通一次就算成功。

更关键的问题是：

- 换一个问题还能不能跑对
- 会不会乱调工具
- 会不会重复走已经完成的步骤
- 信息足够时能不能及时停下来
- 问题不需要工具时能不能直接回答

这就是最基础的稳定性。

所以工程上，我们不能只问：

- 这个 demo 能不能跑

还要问：

- 这个 demo 在不同输入下是否行为一致
- 出错时能不能定位原因
- prompt 是否真的起作用
- 工具描述是否足够清楚

### 4. 什么是最小版本的 Agent 评估

今天你先不用学复杂评测框架。

最小版本的 Agent 评估，只需要做这几件事：

1. 准备几类测试问题
2. 记录每次调用过程
3. 判断结果是否符合预期
4. 分析失败原因
5. 提出优化方案

例如可以把问题分成：

- 不需要工具的问题
- 单工具问题
- 多工具问题
- 长链问题
- 容易诱导模型乱调工具的问题

这已经是很好的工程习惯了。

### 5. Agent 调试的三个关键入口

当 Agent 行为不对时，优先看这三件事：

#### 1. Prompt

重点看：

- 有没有明确告诉模型什么时候不要调用工具
- 有没有说明工具调用顺序
- 有没有强调信息足够时及时停止

#### 2. Tool Definition

重点看：

- 工具描述是否清楚
- 参数名是否易懂
- 工具之间的职责是否重叠

#### 3. Exit Condition

重点看：

- 没有工具调用时是否结束
- 到达 `max_steps` 是否终止
- 失败提示是否足够清晰
